Two different meanings of “one unified query”
There are two reasonable interpretations:
1) One canonical codepath, two stores (what you have now)
One function queries both sources, normalizes results, and returns candidates.
Still two FAISS searches (because there are two indexes).
2) One query execution (what you intended)
This can mean either:
Embed once, reuse the same query vector across both stores (still two FAISS searches, but one embedding call), or
One combined index that contains both Daily and LTM vectors, so there is literally a single FAISS search call returning mixed results (with source metadata).
The practical “best fix” (usually) if you want true unification without merging indexes
If your goal is conceptually one query and to avoid duplicating logic, the best compromise is:
Compute the query embedding once
Run vector search by vector against both FAISS stores (daily + ltm)
Keep the rest identical (candidate normalization, metadata join, ordering, etc.)
This satisfies the spirit of “one unified query” (one embedding) while avoiding the complexity and correctness risk of merging two FAISS stores.
Concretely, it would require changing canonical_retrieve_candidates() to:
call the embeddings model once (OpenAIEmbeddings.embed_query)
use LangChain FAISS’s similarity_search_with_score_by_vector(...) (or equivalent) on both stores
If you truly want one FAISS search (single index)
Then you’d need a combined in-memory index that contains:
all Daily vectors (ephemeral cache)
plus all LTM vectors (loaded from disk)
…and each document would carry metadata.source = daily|ltm (and for daily, the stable daily_entry_id you already added).
This is doable, but it’s heavier:
more memory (duplicating LTM vectors in RAM)
rebuild/invalidation complexity (when uploads index changes, when daily cache changes, embedding model changes, etc.)
