Short answer: yes—most teams use one small LLM call (sometimes two) to build a compact “query bundle” for RAG: route, rewritten standalone query, a few paraphrases, and optional HyDE. Others use rules for routing and a small LLM only for rewriting.

Common implementation patterns
- Single-call builder (most common): One small instruct model returns JSON with route + standalone query + paraphrases + HyDE + entities. Simple and fast.
- Two-stage (router → rewriter): Cheap rule/ML router first; only call the small LLM to rewrite when RAG is needed.
- Rule-first fallback: Regex/NER and heuristics handle 70–90% of turns; small LLM is a fallback for ambiguous queries.
- Offline doc expansion: Use SPLADE/doc2query to expand documents; at query time a tiny LLM just normalizes the user turn.

Typical budgets
- Prompt tokens: 100–300 (includes a 1–2 sentence history summary).
- Output tokens: 50–200 (JSON with 3–5 fields).
- Latency: 50–250 ms on GPU for a 3–8B model; 300–800 ms CPU. Cache aggressively.

Mini model choices
- Open-source 3–8B instruct models (e.g., Llama-3.x-8B, Mistral-7B, Qwen2.5-7B) work well.
- If using provider “mini” endpoints, the smallest reasoning-capable tier usually suffices.
- For reranking, use a dedicated cross-encoder (e.g., bge-reranker, Cohere Rerank) after retrieval.
