# Syx Tools

This folder contains example local tools for querying Syx project memory from an
external agent or shell.

One caveat: this is still an example workflow, not a hardened product. The token
is required by the API shape but not meaningfully enforced yet.

The main example is the agent-memory search bridge introduced by `DELTA-A.4`
and extended by `DELTA-A.5`.

## What This Exercises

The memory search tools call the local backend endpoint:

```text
POST /agent/memory/search
```

That endpoint is read-only. It resolves a human project name, applies the chosen
retrieval category, searches Daily/LTM memory through the retrieval pipeline, and
returns structured JSON snippets. When a result is a bounded Syx memory entry,
the backend may expand it to the full memory entry body.

The tool does not write Syx memory, run sleep, run dream, prune, rebuild indexes,
or generate a chat answer.

## Files

- `agent_memory_search.py` is the Python CLI bridge for the local endpoint.
- `syx_memory_search.sh` is the preferred wrapper for users and agents.
- `config/syx.env` contains optional local defaults for the wrapper.
- `syx_memory_search_skill.md` describes how an agent should use the wrapper.
- `AGENTS.md` is a tiny example instruction file that tells an agent to run a
  memory query and write `chat_summary.md`.
- `CLAUDE.md` points Claude-style agents at `AGENTS.md`.

## Requirements

Before using the wrapper:

1. Start the Syx backend locally, for example with `make run`.
2. Make sure the project you want to search exists in Syx.
3. Set or keep a local `SYX_AGENT_TOKEN` value.

You can inspect available local projects with:

```bash
curl http://127.0.0.1:8000/projects
```

The search endpoint expects `project_name`, not `project_id`.

The `/projects` response includes a `project_names` map. Use the value from that
map as `--project-name`. For example, if the response contains:

```json
{
  "project_names": {
    "004208b5-ebc1-4747-92d5-fb10d2687440": "Main"
  }
}
```

then call the wrapper with:

```bash
--project-name "Main"
```

If a memory search returns `project_not_found`, check `/projects` and retry with
the listed human-readable project name.

## Token Status

`config/syx.env` currently contains:

```bash
SYX_AGENT_TOKEN="local-token"
```

The token is part of the request contract and must be sent. It is intentionally
present now so the tool matches the future secured interface.

At the moment, token authorization is a local-development stub and is not yet
enforced as a real secret. Do not treat `local-token` as production security.
Future work can replace the stubbed authorization boundary without changing this
tool's request shape.

## Basic Usage

From the repository root:

```bash
tools/syx_memory_search.sh \
  --project-name "Main" \
  --query "Find implementation guidance for agent memory search. Context: DELTA-A.4 and DELTA-A.5." \
  --category "SYNTHESIS"
```

You can also call the wrapper by absolute path from another directory. It derives
the repository root from its own location unless `SYX_REPO_DIR` is set.

## Configuration

Optional environment variables:

- `SYX_AGENT_TOKEN`: token sent to the backend; currently required but not
  strongly enforced.
- `SYX_BASE_URL`: backend base URL; defaults to `http://127.0.0.1:8000`.
- `SYX_REPO_DIR`: override repository root; normally not needed.
- `SYX_PYTHON`: override Python executable; normally not needed.

The wrapper uses `venv/bin/python` when it exists, otherwise `python3`.

## Categories

Use the same retrieval categories as the backend route policies:

- `DIRECT` for narrow factual lookup.
- `PROCEDURAL` for implementation, configuration, setup, or execution guidance.
- `EXPLORATORY` for investigation, comparison, or broad discovery.
- `SYNTHESIS` for architecture, design tradeoffs, planning, and idea evolution.
- `OTHER` as a fallback.

For most broad memory searches, start with `SYNTHESIS`.

## Output

The wrapper prints the raw structured endpoint JSON to stdout. Important fields:

- `snippet_count`
- `bounded_result_count`
- `unbounded_result_count`
- `snippets[*].memory_id`
- `snippets[*].semantic_handle`
- `snippets[*].topics`
- `snippets[*].entry_expansion_status`
- `snippets[*].entry_expansion_method`
- `snippets[*].text`

The Python CLI also writes a local debug JSON file under:

```text
tools/agent_interface/
```

That debug output is generated by the CLI and is intended for local inspection.
It should not be committed.

## Agent Example

`tools/AGENTS.md` is a minimal example prompt for another agent:

1. Read `tools/syx_memory_search_skill.md`.
2. Build or use a query from chat history.
3. Run `tools/syx_memory_search.sh`.
4. Summarize the result.
5. Write `chat_summary.md`.

This is intentionally an example workflow, not a fully polished product.

## Troubleshooting

- If you get `project_not_found`, check `/projects` and use the value from the
  `project_names` map as `--project-name`.
- If the wrapper cannot connect, confirm `make run` or the backend process is
  active and `SYX_BASE_URL` points to the correct port.
- If `SYX_AGENT_TOKEN` is missing, set it in your shell or in
  `tools/config/syx.env`.
- If no snippets are returned, try a more retrieval-oriented query with project
  terms, requirement IDs, delta names, file names, or semantic handles.
